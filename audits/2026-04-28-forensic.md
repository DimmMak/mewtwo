# .forensic audit — mewtwo

**Date:** 2026-04-28
**Auditor:** .forensic v0.1.0
**Target:** mewtwo v0.2.0
**Files reviewed:** SKILL.md, CONTRACT.md, FAILURE_MODES.md
**Severity counts:** CRIT=2 · HIGH=3 · MED=3 · LOW=2

---

## Biggest Issues

### 1. **[CRIT] Trigger phrasing has zero activation hooks**
> "description: > The master orchestrator. Receives high-level intent, decomposes into sub-tasks, selects from the skill registry, proposes an execution plan, confirms once, and executes with tiered automation + full audit trail."
> *(SKILL.md:5-10)*

**Why this fails:** The description states WHAT mewtwo is, not WHEN to fire it. Compare snes-builder's frontmatter: *"ALWAYS fire this skill when the user mentions building, making, creating, or scaffolding a skill in ANY phrasing. Triggers include but are not limited to: 'build a skill'..."* — concrete activation phrases. Mewtwo has none. Mechanism: the harness's skill-router needs phrase-matching cues. Without them, mewtwo under-fires; user types "orchestrate this across rumble + journalist" and gets default Claude instead of mewtwo. Self-validating: mewtwo wasn't auto-invoked for this audit despite "audit across multiple skills" being its exact use case.

---

### 2. **[CRIT] Self-reference paradox in contract enforcement**
> "Before calling any skill, verify its registry entry declares: ✓ Reports what it did (no silent success) ✓ Archives what it replaced (no silent delete) ✓ Fails loud with a suggested fix ✓ Reversible (or marked IRREVERSIBLE in registry)"
> *(SKILL.md:121-126)*

**Why this fails:** Mewtwo polices every other skill's contract but has no rule stating it must declare its own contract in its own registry. The orchestrator can violate the four rules without anyone catching it. Mechanism: contract enforcement is unidirectional (mewtwo → others); there is no auditor of the auditor. If mewtwo silently swallows a step's error to avoid stopping the chain, no peer skill detects it. Compare CONTRACT.md:46-60 which declares "fail loud" as universal — yet no clause binds mewtwo to surface its OWN failures.

---

### 3. **[HIGH] NOT-FOR list excludes the most common drift case**
> "NOT for: creating a new skill from scratch (use snes-builder).
>  NOT for: top-level dashboard / routing between domain anchors (use .home).
>  NOT for: fund-specific attention ranking (use .chief)."
> *(SKILL.md:11-13)*

**Why this fails:** The biggest mewtwo-misfire risk is single-skill invocations being routed through mewtwo unnecessarily — e.g., user types `.rumble NVDA` and mewtwo wraps it in a 7-step plan. The NOT-FOR list omits this. Three cases missing: (a) one-shot single-skill calls (`.rumble`, `.price`, `.tier`), (b) read-only queries, (c) skills that already self-orchestrate (royal-rumble's stage 2). Mechanism: ambiguous boundary against single-purpose skills creates ceremony-tax on simple commands.

---

### 4. **[HIGH] "Every failure is loud" with no observable mechanism**
> "Every run is logged. Every write is archived. **Every failure is loud.** Durability-first design."
> *(SKILL.md:9-10)*

**Why this fails:** "Loud" is undefined. No console color, no notification spec, no separate stderr stream, no badge. The STOP banner shown at SKILL.md:144-154 is only loud INSIDE the active conversation. If the user has scrolled away, switched apps, or the task is running unattended, "loud" is silent. Mechanism: claim is asserted but unenforced. Compare F03 recovery (FAILURE_MODES.md:53-67) which describes the user-facing message but ditto — only loud if the user is reading.

---

### 5. **[HIGH] Step 4 is fleet-blocking in current state**
> "If a skill lacks a contract declaration → refuse to call it: 🛑 CONTRACT VIOLATION"
> *(SKILL.md:127-135)*

**Why this fails:** The fleet has 47+ skills. Most do NOT have contract declarations in their registry entries (none of the four skills audited here — royal-rumble, snes-fit, forensic, mewtwo itself — show all four contract checkmarks in any examined file). Strict enforcement means mewtwo refuses to call almost everything. Mechanism: the rule is binary; reality is gradient. Either the rule is silently violated (fail loud rule broken) or mewtwo is functionally dead (orchestrator with empty toolkit). Missing: a "soft-fail" path that warns + proceeds with explicit risk acceptance.

## Other Issues

- **[MED] Subcommand prefix inconsistency** — > "/mewtwo --plan-only [intent]" *(SKILL.md:218)*. Fleet convention is `.skill-name` (`.rumble`, `.tier`, `.forensic`). Mewtwo uses `/mewtwo`. Same skill, two trigger surfaces; user reaches for `.mewtwo` and gets nothing.

- **[MED] Zero worked examples in SKILL.md** — repo has `examples/` dir, but SKILL.md never references a single concrete run. Trigger phrasing has no calibration. Forensic's own SKILL.md:75-86 includes a locked output format; mewtwo provides templates but no completed instances.

- **[MED] RUN_ID format has no timezone or collision guard** — > "Set RUN_ID = mewtwo-{YYYY-MM-DDTHH-MM}" *(SKILL.md:51)*. Two runs in the same minute collide; no UUID suffix or seconds. Single-user system limits blast radius, but the durability claim ("Plain text wins time") implies multi-decade retention where collisions matter.

- **[LOW] CONTRACT.md rule 4 is borderline self-defeating** — > "🟡 ACCEPTABLE (manual reverse documented): transmute:protocol ships a .skill to ~/.claude/skills/. Undo = rm ~/.claude/skills/{name}.skill" *(CONTRACT.md:76-77)*. Allowing "manual reverse documented" to count as reversible weakens rule 4. Either the skill auto-undoes or it's not reversible — the middle tier dilutes the contract.

- **[LOW] FAILURE_MODES.md F08 missing detection trigger** — > "Detection: Pre-execution check for overlapping output paths" *(FAILURE_MODES.md:141)*. The check is named but no implementation pointer (which step, which file, which function). Compare F02 which references the contract checkmarks concretely.

---

## Summary

Mewtwo is a well-specified durability framework whose orchestration logic is sound but whose **trigger surface is silent and whose contract enforcement is suicidal at fleet scale**. The two CRIT issues are the load-bearing failures: nobody fires it (no triggers) and if they did, it would refuse to call most skills (rigid enforcement against a non-compliant fleet). Fix order: (1) rewrite frontmatter description with explicit trigger phrases; (2) add soft-fail path for skills missing contract declarations; (3) extend NOT-FOR list with single-skill / read-only exclusions.
